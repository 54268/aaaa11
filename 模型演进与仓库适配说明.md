# 模型演进与仓库适配说明

## 1. 这份说明是干什么的

这份文档专门说明这套工程不是“从零全 AI 硬编一个开放集 SEI 框架”，而是在你给出的成熟开源仓库基础上做了筛选、适配、整合和本地化重构，再把真正属于你方案创新的部分保留为项目内自实现模块。

重点回答三个问题：

1. 从哪些 GitHub 仓库借鉴或适配了代码
2. 拉下来之后做了哪些修改，为什么不能直接原样用
3. 现在工程里哪些部分是参考成熟模块，哪些部分是围绕你方案自己实现的

## 2. 参考并接入的 GitHub 仓库

当前项目 `third_party/` 下保留了 4 个核心参考仓库：

1. `openmax`
   - 对应仓库：`nikita-petrashen/openmax`
   - 作用：提供 OpenMax 与 Weibull 拟合核心实现

2. `pytorch-ood`
   - 对应仓库：`kkirchheim/pytorch-ood`
   - 作用：作为 OpenMax 的备选实现来源和交叉参考来源

3. `torchcvnn`
   - 对应仓库：`torchcvnn/torchcvnn`
   - 作用：提供复值网络相关组件设计参考，尤其是复值归一化和激活模块

4. `prototypical-networks`
   - 对应仓库：`orobix/Prototypical-Networks-for-Few-shot-Learning-PyTorch`
   - 作用：提供原型损失和原型学习范式的经典参考实现

## 3. 为什么不能直接把第三方仓库原样拼起来

原因主要有四个：

1. 任务不完全一致
   - 这些仓库分别解决的是 OOD 检测、OpenMax、复值网络、ProtoNet 少样本学习。
   - 你的任务是开放集 SEI，并且还包含“关键边界样本挖掘 + 普通边缘样本保留 + 双分支伪未知生成 + OpenMax 融合校准”这一条完整链路。

2. 数据接口不一致
   - WiSig 和 Oracle 的数据组织方式与这些仓库自带示例都不同。
   - 原始仓库代码大多不能直接读你的 SEI 数据格式。

3. 训练目标不一致
   - 你的方案要求已知类表征学习采用：
     - 基础损失
     - 角度损失
     - 原型损失
   - 这和现成仓库中的默认训练脚本并不一一对应。

4. 创新点必须保留为你自己的模块
   - 尤其是关键边界样本定位、竞争边界得分、普通边缘样本与关键边界样本的分流式伪未知生成，这些不能照搬原仓库。

## 4. OpenMax 是怎么适配进来的

### 4.1 直接参考了什么

OpenMax 相关部分主要参考了两个仓库：

1. `third_party/openmax`
   - 直接使用其 OpenMax 与 Weibull 拟合实现思路
   - 当前默认主后端就是它

2. `third_party/pytorch-ood`
   - 作为第二个 OpenMax 后端来源
   - 用来做实现层面的备份和交叉参考

### 4.2 做了什么改造

为了让 OpenMax 能进入你当前的开放集 SEI 主链路，做了这些适配：

1. 增加统一封装层
   - 现在工程里不是训练脚本直接 import 第三方仓库，而是先走本地封装。
   - 这样以后你更换 OpenMax 后端，不需要改主流程。

2. 对接原型分类头输出
   - 你的分类器不是普通 softmax 头，而是 prototype head。
   - 所以要把“原型距离”转换成 OpenMax 可用的激活表示。

3. 把 OpenMax 接进开放集融合校准
   - OpenMax 不再单独决定最终结果。
   - 它输出 `q_om`，再和原型距离未知分数 `q_pd` 融合。

4. 适配保存与恢复
   - 第三方原实现通常不是按你这套工程的方式保存中间状态。
   - 现在已经统一为本地 `state_dict / pickle / json` 形式，方便串联。

### 4.3 为什么这不算“原样照抄”

因为真正最终的开放集决策逻辑已经不是第三方仓库原来的默认脚本，而是：

1. 先做你的已知类原型表征学习
2. 再做你的边界挖掘和伪未知生成
3. 再用 OpenMax 做一条未知分数
4. 再和原型距离未知分数融合
5. 最后按你当前目标做阈值选择

也就是说，OpenMax 是被吸收到你的方案链路里，而不是你的方案变成了 OpenMax 示例代码。

## 5. CVCNN 是怎么适配进来的

### 5.1 直接参考了什么

复值网络部分参考了 `torchcvnn` 仓库。

重点借鉴的是：

1. 复值张量在网络中的组织方式
2. 复值归一化与激活的设计思路
3. 复值神经网络模块的组合方式

### 5.2 为什么没有完全原样直接用 `torchcvnn`

主要原因是工程环境和任务形式不完全适配：

1. 你当前工程是 1D I/Q 序列
   - 而 `torchcvnn` 更偏通用复值模块库，不是专门给你这条 1D SEI 主链直接配好的。

2. 当前环境兼容性有限
   - 在你当前本机环境下，直接全量依赖上游模块并不稳定。
   - 所以项目里保留了第三方仓库，同时实现了本地可稳定运行的 `cvcnn_iq` 主干。

3. 你的训练链路需要和原型头、边界模块、OpenMax 校准无缝衔接
   - 直接照搬第三方模型结构反而会增加改造成本。

### 5.3 最终怎么处理

最终采取的是“双轨方式”：

1. 保留 `third_party/torchcvnn/`
   - 证明复值网络设计确实参考了成熟开源仓库

2. 保留 `torchcvnn_hybrid.py`
   - 这是直接调用 `torchcvnn` 复值模块的适配版本

3. 当前默认主干使用 `cvcnn_iq.py`
   - 它是围绕你当前 1D I/Q 开放集 SEI 主链做的本地稳定版 CVCNN
   - 逻辑上仍然是 CVCNN，不是回退到普通实值 CNN

## 6. ProtoNet 是怎么参考进来的

### 6.1 直接参考了什么

原型学习部分主要参考了 `prototypical-networks` 仓库，重点参考：

1. 原型由类内样本均值构成
2. support/query 的原型损失计算逻辑
3. `prototypical_loss.py` 的基本实现方式

### 6.2 做了什么适配

你的方案不是标准 few-shot episodic learning，所以没有把该仓库的整套训练脚本原样搬进来，而是做了下面两步：

1. 主分类头改为本地 `prototype_head`
   - 负责日常训练和推理中的原型分类

2. 把 ProtoNet 的 loss 做成可选正则项
   - 通过本地 `EpisodicProtoRegularizer` 包装第三方 `prototypical_loss`
   - 当前默认配置下，这个项是关闭的
   - 因为你当前方案的主损失已经明确是三部分：
     - 基础损失
     - 角度损失
     - 原型损失

### 6.3 现在和你方案的关系

也就是说，ProtoNet 在这套工程中更多是“理论与实现参考来源”，而不是训练主流程的原封不动搬运。

## 7. 你方案里真正自己实现的部分

下面这些部分是围绕你的方案自己实现的，不是第三方仓库直接给的：

1. 边界挖掘模块
   - 关键边界样本定位
   - 普通边缘样本保留
   - 竞争边界得分计算

2. 伪未知生成模块
   - 普通边缘样本沿本类外法向外推
   - 关键边界样本按“本类外法向 + 最近异类原型排斥方向”联合生成

3. 原型距离未知分数
   - 不只依赖 OpenMax
   - 额外增加一条原型距离未知分数支路

4. 融合校准
   - `q_om + q_pd` 融合
   - 并且已经发展出“已知类优先”的校准策略

5. 数据集适配
   - WiSig 四子集统一预处理
   - Oracle SigMF 解析
   - 原始标签保存与开放度曲线绘图支持

## 8. 这套工程是怎么一步步演进到现在的

### 阶段 1：先把最小可跑通链路搭起来

这一阶段的目标不是最好结果，而是先把完整流程跑通：

1. 已知类训练
2. 边界挖掘
3. 伪未知生成
4. OpenMax 拟合
5. 融合校准
6. 开放集评估

### 阶段 2：把 backbone 对齐到你的方案

一开始工程里保留过基础 I/Q CNN 作为兜底。

后来逐步收敛到：

1. 默认 backbone 改为 CVCNN
2. 三段损失明确化
3. WiSig / Oracle / demo 配置统一到相同训练逻辑

### 阶段 3：解决“已知类被过度误拒”的问题

最开始 Oracle 的一个明显问题是：

1. 闭集分类本身不差
2. 但开放集阶段对已知类误拒过多

所以后面又演进出：

1. 融合参数搜索
2. 已知类优先的校准目标
3. 更细的阈值搜索

### 阶段 4：把 WiSig 四子集接进来

为了避免只在 Oracle 上好看，又增加了：

1. WiSig 四子集联合预处理
2. 去重逻辑
3. 跨子集平衡协议
4. 每样本归一化

### 阶段 5：补全图表与结果展示

为了让结果更适合方案验证和后续写作，又补上了：

1. ROC 曲线
2. PR 曲线
3. 混淆矩阵图
4. 未知分数分布图
5. Openness–Metric 曲线
6. 精简版指标总览

## 9. 总结

可以把当前工程理解成下面这个关系：

1. 成熟仓库负责提供“可复用基础能力”
   - OpenMax
   - ProtoNet loss
   - 复值网络模块设计

2. 本地工程负责把这些能力转成“开放集 SEI 方案链路”
   - 数据集适配
   - backbone 与原型头对接
   - 关键边界样本与普通边缘样本分流
   - 伪未知生成
   - 多分数融合校准
   - 图表与评估输出

3. 你方案的创新点仍然保留在本地模块里
   - 没有被第三方仓库覆盖掉

所以这套工程可以明确说是：

- 参考了成熟 GitHub 仓库
- 进行了真实的适配和改造
- 不是简单拼接，更不是整套原样照搬
- 最终目标是服务于你的开放集 SEI 方案验证
